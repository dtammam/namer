# Versioning & Release Policy

`namer` follows [Semantic Versioning](https://semver.org) — `MAJOR.MINOR.PATCH`.

## What each version component means for namer

| Component | Increment when… |
|-----------|----------------|
| **MAJOR** | Breaking CLI changes: a flag is renamed or removed, output format changes in a way that breaks scripts |
| **MINOR** | New features: new flags, expanded word lists, new output modes |
| **PATCH** | Bug fixes, documentation-only changes, word list corrections |

## Release checklist

1. Update the `version` field in `Cargo.toml`.
2. Write user-facing PR titles (these become release notes — see below).
3. Tag the release commit: `git tag v<version>` (e.g., `v0.2.0`).
4. Push the tag: `git push origin v<version>`.
5. Verify CI passes on the tag.
6. Verify the GitHub Release is created with auto-generated release notes.

## Release notes

Release notes are auto-generated by GitHub from the titles of PRs merged since
the previous tag. This is configured via `generate_release_notes: true` in the
release workflow (`.github/workflows/release.yml`). No manual release notes
file is needed.

The quality of release notes depends entirely on PR title quality — see
**PR title guidance** below.

## PR title guidance

GitHub auto-generates release notes from PR titles. Write every PR title from
a user's perspective, not an implementer's perspective.

Good: `Add --number flag for bulk name generation`
Bad: `impl number flag`

Good: `Expand adjective and noun lists to 777 entries each`
Bad: `update words.rs`

If the PR title wouldn't make sense in a user-facing changelog, rewrite it
before merging.
