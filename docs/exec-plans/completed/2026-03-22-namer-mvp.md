# Namer MVP

## Goal

Produce a CLI binary that prints one random `adjective-noun` pair to stdout and
exits cleanly, with word lists hardcoded in source.

## Scope

- A single Rust binary named `namer`
- Invoked with no arguments
- Selects one adjective at random from a hardcoded list
- Selects one noun at random from a hardcoded list
- Prints the pair to stdout as `<adjective>-<noun>` (lowercase, hyphen-separated)
- Exits with code 0
- Word lists are `const` arrays compiled into the binary
- Total implementation in `src/main.rs` only, targeting under 50 lines
- Every public function has explicit unit tests covering its behavior

## Out of scope

- Flags of any kind (`--count`, `--separator`, `--style`, `--alliterative`, etc.)
- Config files
- External word list files
- Multiple output lines
- Custom separators
- Named generation styles beyond the single default adjective-noun pair
- Library/public API surface
- CI pipeline (roadmap item R8, not MVP)
- Any runtime dependency beyond `std`, `clap`, and `rand` (note: MVP does not
  require clap given there are no flags; rand is the only expected addition)
- HTTP, web UI, or any networked interface
- Uniqueness guarantees across invocations

## Constraints

- Language: Rust
- No `unsafe` blocks
- No `unwrap()` in library code; use `expect()` with context if unwrap is needed
- `cargo fmt -- --check` must pass
- `cargo clippy -- -D warnings` must pass (zero warnings)
- Runtime dependencies: `std`, `clap`, `rand` only (CLAUDE.md non-negotiable)
- Binary name must be `namer`
- Implementation must stay under 50 lines of Rust

## Acceptance criteria

- [x] Running `namer` (no arguments) prints exactly one line to stdout
- [x] The output line matches the pattern `<word>-<word>` — two lowercase
      alphabetic tokens separated by a single hyphen, with no leading or trailing
      whitespace beyond a trailing newline
- [x] The first token is drawn from the adjective word list hardcoded in source
- [x] The second token is drawn from the noun word list hardcoded in source
- [x] The process exits with code 0
- [x] Nothing is written to stderr on a clean invocation
- [x] The adjective and noun lists each contain at least 10 entries, giving a
      minimum combinatorial space of 100 unique pairs
- [x] Output varies across multiple invocations (non-deterministic; verified by
      running `namer` at least 5 times and observing at least 2 distinct outputs)
- [x] `cargo test` passes with no failing tests
- [x] `cargo build` produces the binary without errors or warnings
- [x] `cargo clippy -- -D warnings` passes with zero warnings
- [x] `cargo fmt -- --check` passes
- [x] The `src/main.rs` implementation does not exceed 50 lines

## Design

### Approach

The MVP is a single-file Rust binary (`src/main.rs`) that selects one random
adjective and one random noun from hardcoded `const` arrays and prints them
hyphen-separated to stdout. The only non-std dependency is `rand`.

Given the 50-line budget, the file will contain: two `const` arrays (adjectives
and nouns, at least 10 entries each), one public helper function
`generate_name()` that accepts an `&mut impl Rng` and returns a `String`, and a
`main()` that seeds a thread-local RNG, calls the helper, and prints the result.
Injecting the RNG via a trait parameter makes the helper deterministically
testable without mocking or process-level invocation.

No `clap` dependency is needed since there are no CLI flags.

### Component changes

- **`Cargo.toml`**: Add `rand = "0.9"` under `[dependencies]`. No other changes.
- **`src/main.rs`**: Replace the current hello-world stub with the full
  implementation described below.

#### `src/main.rs` structure (top to bottom)

1. `use rand::Rng;` -- single import.
2. `const ADJECTIVES: &[&str] = &[...];` -- at least 10 lowercase alphabetic
   words. No hyphens, no digits, no uppercase.
3. `const NOUNS: &[&str] = &[...];` -- same rules, at least 10 entries.
4. `pub fn generate_name(rng: &mut impl Rng) -> String` -- picks a random index
   into each array using `rng.random_range(0..len)`, formats
   `"{adjective}-{noun}"`, returns the `String`. Doc comment required.
5. `fn main()` -- creates `rand::rng()`, calls `generate_name`, prints via
   `println!`. No error paths needed (infallible).
6. `#[cfg(test)] mod tests` -- unit tests for `generate_name`.

Estimated line count: approximately 40-45 lines (including doc comments, blank
lines, and the test module). This leaves a small margin under the 50-line cap.

### Data model

Two compile-time constant slices of string literals:

- `ADJECTIVES: &[&str]` -- curated list of 10+ common English adjectives, all
  lowercase alphabetic (e.g., "quick", "bright", "calm", "bold", "eager",
  "fierce", "gentle", "happy", "keen", "lively", "mighty", "noble").
- `NOUNS: &[&str]` -- curated list of 10+ common English nouns, all lowercase
  alphabetic (e.g., "fox", "river", "cloud", "falcon", "grove", "harbor",
  "island", "jaguar", "koala", "lantern", "meadow", "nova").

No runtime data structures, no files, no deserialization.

### API changes

None. This is a CLI binary with no library surface. The only public function
(`generate_name`) is public for testability within the crate, not as an
external API.

### Test strategy

All tests live in `#[cfg(test)] mod tests` inside `src/main.rs`.

The key design decision is that `generate_name` takes `&mut impl Rng`, which
allows tests to supply a seeded deterministic RNG (`rand::rngs::mock::StepRng`
or `rand::SeedableRng` with a fixed seed). This avoids flaky tests.

Required tests:

1. **`test_generate_name_format`** -- call `generate_name` with a seeded RNG,
   assert the result matches the regex pattern `^[a-z]+-[a-z]+$` (or
   equivalently, split on `-` and verify exactly two parts, both non-empty and
   all-alphabetic).
2. **`test_generate_name_from_word_lists`** -- call `generate_name` with a
   seeded RNG, split the output on `-`, and verify the adjective is in
   `ADJECTIVES` and the noun is in `NOUNS`.
3. **`test_generate_name_deterministic`** -- call `generate_name` twice with
   identically-seeded RNGs, assert both return the same string. This proves
   the function is purely a function of RNG state.

Note: the `StepRng` constructor is `StepRng::new(initial, increment)`. Using
a fixed initial value and increment of 1 gives predictable index selection.
The developer should verify which concrete type is available in `rand 0.9`
(`rand::rngs::mock::StepRng` or the equivalent) and use it accordingly.

These three tests cover: output format correctness, word-list membership, and
determinism given identical RNG state. Together they exercise the single public
function completely.

### Alternatives considered

**Alternative A: No RNG injection -- test via process invocation.**
Run the compiled binary multiple times in integration tests and regex-match
stdout. Pros: no need for a public function; `main` is the only entry point.
Cons: slower tests (subprocess spawn), flaky assertions about randomness,
cannot verify word-list membership deterministically, harder to stay under 50
lines with both integration test infrastructure and the implementation. Rejected
because it produces weaker test coverage and is slower.

**Alternative B: Split into `src/lib.rs` + `src/main.rs`.**
Put word lists and `generate_name` in a library crate, keep `main.rs` as a thin
entry point. Pros: cleaner separation, enables reuse. Cons: the exec plan
explicitly scopes to "implementation in `src/main.rs` only" and puts a library
API surface out of scope. Rejected because it violates the stated scope.

### Risks and mitigations

- **Risk**: The 50-line limit is tight once doc comments, test module, and word
  lists are included. -> **Mitigation**: The word list arrays can be written as
  single-line comma-separated literals. Doc comments can be single-line `///`.
  The developer should count lines before committing and trim if needed. If
  truly tight, reduce word lists to exactly 10 entries each.

- **Risk**: `rand 0.9` API surface may differ from `0.8` (e.g., method names
  like `gen_range` vs `random_range`). -> **Mitigation**: The developer should
  check `rand 0.9` docs. In 0.9, the method is `random_range()` on the `Rng`
  trait. If the API has changed, adapt accordingly -- the design is not coupled
  to a specific method name.

- **Risk**: `rand 0.9` with `edition = "2024"` may have MSRV or compatibility
  issues. -> **Mitigation**: Rust 1.94 is very recent; `rand 0.9` supports it.
  The build-specialist agent will catch any compilation failure immediately.

### Performance impact

No expected impact on performance budgets. Adding `rand` as the sole dependency
is well within the < 10s build time budget. The test suite will contain 3 unit
tests, far under the < 10s test budget.
